## 已修复（本次提交）✅

1. **create_order 签名回归（致命）**：工具层仍按旧签名传 `string[]`，而 store.createOrder 已改为 `[{id, qty}]`——AI 下单会创建空订单。已适配。
2. **高风险无证据兜底文案危险（安全）**：原硬编码"请先少量多次补水并密切观察"，对尿闭、中毒等急症是危险指引。已改为不给任何居家处置、直接就医引导。
3. **知识 Agent 模型调用无保护（安全）**：deepseekChat 异常直接抛出、JSON 解析失败默认 `canAnswer=true`。已改为：调用失败/解析失败一律走保守拒答。
4. **引用背书造假（可信度）**：白名单权威站点只是静态目录映射、从未抓取原文，但输出会拼"参考：Merck《…》"，等于虚构支撑关系。已改为：只有真正读过内容的内部文章用"参考"，白名单来源降为"可进一步查阅"。
5. **高危词表误伤**：单字"药"、高频词"马上/立刻"会把"马上双十一了""狗粮里能拌营养膏吗"误判为高风险并硬阻断导购。已收窄为组合词，并补充抽搐/昏迷/休克等真急症词。
6. **history 角色未过滤（安全）**：客户端请求体里的消息未做角色白名单，可注入伪造 system 消息。已在 runAgentCore 入口过滤为 user/assistant。
7. **ask_knowledge_agent 丢参**：工具 schema 声明了 conversationContext/suspectedRiskTags，执行时却硬编码为空数组，削弱多轮风险判定。已透传。
8. **转人工信号接通**：知识 Agent 高风险时 needsHumanHandoff=true，并透传到展示策略层（原来恒 false）。

## 建议原作者跟进 ⬜

按优先级：

1. **守门误伤问题（体验）**：guards.ts 大量依赖正则/全等字符串匹配，引用格式措辞稍变即拒绝、常见导购句式被误判缺货、澄清式追问被"proposals≥1"强制封死。每次守门拒绝消耗一个 step（上限 8），叠加后正常请求容易被推到兜底话术。建议：规则误判时守门降级为"提示"而非"硬拒"。
2. **幻觉商品 id 未校验（正确性）**：searchedProducts 传入守门却未校验 proposals 里的商品 id 是否真实存在，模型编造的 id 会全数放行到前端（前端会静默丢弃，用户看到空方案）。
3. **高风险状态无粘性（安全）**：路由只看最后一条消息，用户上一轮说"疑似中毒"、下一轮说"推荐点吃的"即可绕过阻断。建议风险标签跨轮保留。
4. **request.ts 贪婪抽词**：典型导购问句会被抽出过宽的"明确商品词"导致整链失败、错误回复"商品未上架"；且工具执行时用启发式 ctx 值静默覆盖模型参数，模型无法自我纠偏。
5. **runAgentStream 是伪流式死代码**：事件全量缓冲、跑完才吐，且无任何路由使用、绕过额度。要么接真流式（deepseek stream:true + BFF SSE），要么删除。
6. **额度交互回归**：runAgentCore 全量 try/catch 永不抛错，services.ts"模型故障不扣额度"的设计失效（现在故障也扣）。建议失败时返回带标记的结果或抛出特定错误。
7. **成本知悉**：单次对话最多 8 步模型调用 + 知识 Agent 独立调用 + 守门重试，额度只计 1 次——token 成本与延迟显著放大，上线前建议压测与设上限告警。
8. **分层**：server/agent 反向 import components/data（前端层）；文章数据建议迁到 lib/ 共享层（与 catalog.ts 同级）或入库。

## 总体评价

设计理念（规则路由降低模型自由度、证据白名单收口、风险分级由本地规则而非模型判定、守门可重试）都是对的，和团队目标架构图高度吻合。主要短板是"规则的精度撑不起硬阻断的权力"——关键词/正则误判被守门放大成确定性错误行为。方向建议：守门对"安全类"保持硬（宁可误伤），对"体验类"（引用格式、句式、方案数量）降为软提示。
