# Pawly（宝莉）从 Demo 到上线：公司化 · 微信小程序 · App 全实现路径

> 版本：v1.1　更新日期：2026-07-02
> 目标：把现有的 Next.js 全栈 Demo（AI 宠物导购电商）推进到「成立公司 → 微信小程序正式上线 → 原生 App 上线」的完整商业化产品，并给出 6 人团队的分工建议。整体周期已按要求压缩至 **6 个月**。

---

## 一、总体路线图（时间线概览，已压缩至 6 个月）

按要求将整体周期压缩到 **6 个月**。压缩的核心手段是把原本"串行"的阶段最大程度**并行化**——公司注册、各类工信部/网信办备案、技术改造、小程序开发、App 开发提前重叠启动，而不是等上一阶段完全结束再开始下一阶段。需要注意：**生成式人工智能算法备案等审批环节的周期由主管部门决定，不受团队并行度影响**，是压缩计划中唯一不能靠"加人加班"缩短的硬约束，6 个月是较激进但可行的目标，存在因备案审批延迟滞后至 7-8 个月的风险（见第七章风险提示）。

| 阶段 | 建议周期 | 核心目标 | 关键产出 |
|---|---|---|---|
| **Phase 0** 公司与合规筹备 | 第 1-3 周 | 完成主体注册、基础资质、商标域名保护，同步启动所有长周期备案 | 营业执照、公户、ICP 备案提交、商标申请、生成式AI备案提交 |
| **Phase 1** 产品工程化改造 | 第 1-8 周（与 Phase 0/2 并行） | 把 Demo 改造为可承载真实交易与真实用户的系统 | 真实登录、真实支付沙箱、供应链对接方案、AI 安全护栏 |
| **Phase 2** 微信小程序上线 | 第 6-14 周（约第 2 月末-第 3 月） | 小程序过审上线，完成冷启动 | 小程序备案+过审、微信支付商户号、首批种子用户 |
| **Phase 3** 数据沉淀 + App 并行开发 | 第 3-5 月 | 用真实数据验证 AI 导购转化率；App 开发与小程序运营同步推进，不等小程序数据完全跑出来再启动 | 复购率/转化率数据、生成式AI备案完成、App 开发主体完成 |
| **Phase 4** App 上线 | 第 5-6 月 | iOS/Android 双端上架 | App Store + 主流安卓应用市场上架 |
| **Phase 5** 规模化运营/融资 | 第 6 月起 | 规模化获客、探索自有供应链/融资 | 增长曲线、BP、下一轮资质（如食品经营） |

> **关键提醒**：生成式人工智能算法备案、增值电信业务许可证（如需要）、宠物食品类资质，审批周期普遍在 1-3 个月甚至更长，且不可加急，必须在**第 1 周**就提交，不能等技术做完再申请，否则会成为压缩到 6 个月计划里最大的风险点。

## 二、法律合规与公司成立

### 2.1 公司主体注册
- **公司类型**：有限责任公司；建议创始团队提前约定股权结构，预留 10-20% 期权池给后续核心成员/融资。
- **经营范围**：至少包含「宠物用品、宠物食品（预留）零售」「技术开发、技术服务、技术咨询」「互联网信息服务」「网络技术研发」，为后续拓展预留空间，避免频繁变更经营范围。
- **注册资本**：若明确会申请增值电信业务经营许可证（ICP 许可证，见 2.2），注册资本建议设置在 **100 万元以上**（外资另有 1000 万元门槛），并注意实缴/认缴要求。
- **配套动作**：对公银行账户、税务登记（一般纳税人/小规模纳税人选择）、社保公积金开户（团队全职化后合规用工的前提）。
- **知识产权先行**：尽快申请「Pawly」「宝莉」商标（多类别，尤其第 35 类广告销售、第 9/42 类软件、第 31 类宠物食品），锁定小程序昵称、App Store 应用名、域名，防止被抢注。
- **软件著作权**：为核心系统（Agent 编排系统、商城系统）申请软著，用于 App 上架材料、未来融资/招投标尽调。

### 2.1.1 问答：成立「一人公司」能否缩短进程？是否还能正常做工信部备案？

**结论：能小幅节省注册环节的时间，但对整体 6 个月节奏影响很小；且完全不影响后续 ICP 备案、生成式AI算法备案等工信部/网信办备案的正常推进——这些审批看的是「企业材料是否齐全」，与股东是一人还是多人无关。** 不建议仅为了赶进度而长期采用一人公司架构，理由如下：

- **能省的时间有限**：一人公司不需要起草、协商、签署多方股东协议和公司章程，跳过股东间意见对齐环节，理论上比多股东公司快 3-7 个工作日拿到营业执照。但工商注册本身（无论一人或多股东）走「一网通办」通常 3-15 个工作日即可完成，**本来就不是本计划的时间瓶颈**，真正卡时间的是下面几类备案。
- **各类备案不受股东人数影响，可以正常并行推进**：ICP 备案、生成式人工智能算法备案、增值电信业务许可证、小程序备案、宠物食品资质等，审核主体是「企业」本身（材料是否齐全、安全评估/技术能力是否达标），**一人公司可以同步正常申请，不会被卡住**。
- **一人公司的隐性代价（建议只作过渡方案，不长期采用）**：
  1. 《公司法》要求一人有限责任公司每年年终编制财务会计报告并经会计师事务所审计，否则股东对公司债务承担连带责任——等于**每年多一笔强制审计费和合规动作**；
  2. 本项目是 6 人共同创业，一人公司必须先完成「增资扩股 + 公司类型变更」（一人有限 → 多股东有限责任公司）才能给其他 5 人发放股权/期权，这个变更本身也要走 1-2 周的工商变更流程，等于把时间成本挪到了后面，并没有真正省下来；
  3. 融资尽调时，股权结构过于集中会被投资人要求先规范化，反而可能拖慢融资节奏。
- **建议做法**：由主要发起人先成立一人公司作为过渡壳公司，**第 1 周立即用它启动 ICP 备案、生成式AI备案等不依赖股权结构的流程**（不耽误进度），同时团队尽快签署合伙人协议明确权益比例，在第 1-2 个月内完成增资扩股，把公司变更为 6 人（或核心成员）持股的有限责任公司。这样既不牺牲注册速度，也避免后续二次变更的隐性时间成本和法律风险。

### 2.2 互联网信息服务类合规
- **ICP 备案**（非经营性，免费）：是小程序、App、官网上线的**前置条件**，苹果 App Store 中国区、各安卓应用商店均已强制要求提供 ICP 备案号。需在服务器/域名确定后尽早提交。
- **增值电信业务经营许可证（ICP 许可证/EDI 证）**：如果通过平台**直接向用户收费**（如会员订阅、平台佣金结算）或以电商方式经营性变现，理论上需要办理。若早期采用「导购+跳转第三方平台下单」模式可暂缓，但**自建交易闭环（应用内下单+微信支付）大概率需要**，建议 Phase 0 同步评估并启动办理（对注册资本、技术团队人数、经营场所有硬性要求，年审）。
- **公安网安备案**：与 ICP 备案并行，向属地网安部门办理。

### 2.3 生成式人工智能合规（本项目的关键合规项 ⭐）
Pawly 的 AI 客服本质上是「面向境内公众提供的生成式人工智能服务」，需要按《生成式人工智能服务管理暂行办法》履行算法备案：
- 提前准备材料：产品基本情况、模型来源说明（自研 or 调用 DeepSeek 等第三方 API 需明确责任划分）、语料/训练数据合规声明、安全评估报告、内容安全防范机制说明、用户投诉/内容标注机制。
- **建议委托专业算法备案代办机构**协助（周期通常 1-3 个月，官方无法加急），Phase 0 立项时同步启动。
- **配套技术改造**（详见第三章）：敏感词/违规内容过滤、宠物医疗类免责声明强化（AI 不得做出诊断性结论）、异常对话拦截并转人工。

### 2.4 宠物用品/食品经营资质
- **纯用品类（玩具、日用品、非食品）**：营业执照经营范围覆盖即可开始销售，无需额外前置许可，合规成本最低，**建议作为第一阶段主推品类**。
- **宠物食品/保健品类**：
  - 国产：需供应商侧持有《饲料生产许可证》《饲料添加剂生产许可证》及产品批准文号；
  - 进口：需《进境动植物检疫许可证》、报关备案、CMA/CNAS 质检报告；
  - **建议策略**：早期不做自营囤货，采用「联营分销」模式对接已持证的供应商/京东联盟/淘宝联盟等渠道，把食品资质压力转移给上游持证方，自己只承担平台/导购角色；自建宠物食品品牌放到 Phase 4+ 之后、有资金和团队规模时再评估。

### 2.5 数据安全与个人信息保护（PIPL）
- 用户的宠物档案、聊天记录属于个人信息（部分可能涉及敏感个人信息），需要：
  - 明确的《隐私政策》《用户协议》，首次使用需弹窗明示同意（当前 Demo 用匿名 Cookie 静默建档，**上线前必须补充授权流程**）；
  - 数据最小化与留存期限设计；
  - 若第三方大模型 API 服务器在境外或涉及跨境传输，需评估《个人信息保护法》第 38 条数据出境合规要求，优先选择境内可备案的模型供应商/私有化部署方案；
  - 建议上线前做一次「App/小程序个人信息保护自评」，参照工信部相关检测规范，避免被要求整改下架。

### 2.6 微信小程序专项合规
- **小程序备案**：2023.9.1 起强制要求，流程为「注册 → 认证 → 备案」，含工信部短信核验 + 通管局审核，平台初审 1-2 个工作日，整体建议预留 1-2 周；每年需完成年审。
- **类目选择与资质**：一个小程序最多选 5 个服务类目，每月 5 次修改机会；电商类目按品类需提交对应资质（食品类需额外资质，前期建议选择「日用百货/宠物用品」等非食品类目降低门槛）。
- **微信支付商户号**：需公司营业执照+对公账户，公司注册完成后即可申请，建议与小程序开发并行推进。

### 2.7 App 专项合规
- iOS App Store 中国区已强制要求提供有效 ICP 备案号，安卓主流应用商店（华为、小米、OPPO、vivo 等）同样要求。
- 企业开发者账号：Apple Developer Program（企业级，299 美元/年）+ 各安卓应用商店企业开发者认证（需营业执照）。
- 上架材料：软件著作权证书（或电子版权证书）、ICP 备案截图、隐私政策链接、（如涉及收费）ICP 许可证等。

## 三、技术改造路径（从 Demo 到生产级）

当前代码库（`app/`、`lib/`、`components/`）是一个功能完整但**单机 Demo 级**的实现：匿名 Cookie 会话、单一 DeepSeek 模型、静态种子商品数据、无支付、无真实供应链、无内容安全护栏、无测试与 CI。上线前需要在以下 5 个模块补齐：

### 3.1 AI Agent 层（对应现有 `lib/agent/`）
- [ ] **内容安全护栏**：Prompt Injection 防御、敏感/违规内容过滤、宠物医疗类问题的免责声明强制升级（AI 不得给出诊断性结论，需引导至"请咨询执业兽医"）
- [ ] **转人工兜底机制**：当前完全没有人工客服兜底通道，需要设计"AI 识别异常/投诉/超纲问题 → 转人工"的流程
- [ ] **多模型容灾**：目前强依赖单一 DeepSeek API，建议接入至少一个备用模型（如 Qwen/GLM/Kimi）做 Failover，降低单点风险和议价风险
- [ ] **RAG 知识库**：现有做法是把商品检索结果动态塞进上下文，规模变大后需要引入向量检索（宠物科普、商品详情、FAQ），控制 token 成本
- [ ] **成本与效果监控**：token 用量看板、推荐转化率埋点、A/B 测试框架，用数据持续优化 Prompt 和工具设计

### 3.2 后端与基础设施（对应现有 `lib/db/`、`app/api/`）
- [ ] **用户体系升级**：匿名 Cookie → 微信授权登录（小程序天然具备 `wx.login`）+ 手机号绑定，保留匿名试用但登录后数据合并
- [ ] **真实支付**：接入微信支付（小程序）与未来支付宝，设计完整订单状态机（待支付/已支付/已发货/已完成/退款中/已退款）
- [ ] **真实供应链对接**：替换 `lib/db/seed.ts` 静态数据，对接京东联盟/淘宝联盟 API 或宠物垂类供应商接口，实现库存、价格、物流实时同步
- [ ] **订单履约系统**：物流对接（快递鸟/快递 100 等）、库存扣减与超卖防护、售后/退款流程
- [ ] **安全加固**：API 限流、WAF、审计日志、手机号/地址等敏感字段加密存储
- [ ] **可观测性**：接入日志与监控告警（如 Sentry）、压测预案
- [ ] **数据库升级**：从 Neon 免费实例的 Demo 配置升级到生产级容量规划 + 定期备份策略

### 3.3 微信小程序端
- [ ] 技术选型：建议用 **Taro/uni-app** 等跨端框架改造现有 React 组件（`components/`），实现小程序与未来 App 的代码最大化复用，而非小程序原生重写
- [ ] 小程序专属能力：微信授权登录、微信支付、订阅消息（物流/客服提醒）、分享裂变（拼团/分享有礼，契合社交冷启动）

### 3.4 App 端（iOS + Android）
- [ ] 技术选型：React Native / Flutter（复用团队 React 经验）或直接复用 uni-app 产物，一套代码多端输出
- [ ] App 专属能力：推送通知、深链接、离线缓存、启动性能优化
- [ ] 上架材料准备（软著、隐私政策、ICP 备案、企业开发者账号，见 2.7）

### 3.5 工程化基建
- [ ] 从 0 搭建 CI/CD（当前仓库无 `.github/workflows`）
- [ ] 补充自动化测试（当前项目暂无测试覆盖，需要至少覆盖 Agent 工具调用、订单/支付关键链路）
- [ ] 灰度发布与回滚机制

## 四、产品与商业化建议

- **盈利模式**：早期建议以「导购 + 联盟佣金」为主（对接京东联盟/淘宝联盟等，赚取分佣），验证 AI 导购的转化率和复购率后，再逐步引入自营高毛利品类或订阅会员模式，降低早期库存与资质压力。
- **冷启动打法**：微信生态适合社群/社交裂变（宠物主社群运营、KOC 测评种草、视频号内容导流至小程序），比纯 App 冷启动成本更低，建议 Phase 2 优先把小程序作为主战场。
- **核心壁垒**：AI 沉淀的宠物档案数据反哺选品与个性化推荐是长期护城河，需要提前设计好数据合规使用边界（哪些数据可用于推荐算法训练、需要哪些用户授权）。

## 五、六人团队分工建议

按照目前的技术栈（Next.js 全栈 + AI Agent + 未来小程序/App 多端）和上线所需的合规/市场工作，建议如下分工。早期 6 人团队角色会有交叉，法务/代办类工作建议外包给专业机构，团队内部主要承担"对接与决策"角色，而非亲自跑流程。

| 角色 | 姓名/占位 | 核心职责 | 关键交付物 |
|---|---|---|---|
| **① 产品负责人**（可由 CEO 兼任） | — | 产品路线图定义、公司注册与法务合规统筹（含一人公司过渡→多股东变更节奏把控）、对外融资/BP、跨角色协调；6 个月压缩节奏下承担"排期总控"角色 | 产品路线图、PRD、公司资质清单、BP |
| **② 产品设计师（UX/UI）** | — | 用户研究、信息架构、小程序/App 交互与视觉设计、AI 对话式交互体验设计（配合③打磨 ChatWidget 的引导与推荐呈现） | 高保真原型、设计规范、可用性测试报告 |
| **③ AI/Agent 架构师** | — | Agent 编排与 Prompt 设计（`lib/agent/`）、模型选型与成本控制、RAG 知识库、AI 内容安全护栏、生成式AI备案材料技术支撑 | Agent 工具集迭代、安全护栏方案、备案技术说明文档 |
| **④ 后端/基础设施工程师** | — | 数据库与 API（`lib/db/`、`app/api/`）、支付与订单履约、供应链对接、安全加固、CI/CD | 生产级后端系统、支付闭环、监控告警体系 |
| **⑤ 前端/多端工程师** | — | 小程序与 App 前端工程实现（复用 `components/`，落地②的设计稿）、性能优化、跨端适配 | 小程序上线包、App 上线包 |
| **⑥ 市场运营/商务 BD** | — | 市场与竞品分析、供应链谈判（联盟平台/供应商）、冷启动获客与内容运营、客服团队搭建、资质办理执行对接（配合外部代办机构） | 市场分析报告、供应链合作协议、冷启动增长数据、资质进度看板 |

**协作建议**：
- ①②③④⑤ 五人构成产品/设计/技术闭环，建议每周固定同步 Agent 效果数据（转化率、差评/投诉）与工程排期；6 个月压缩节奏下建议改为每周两次站会，避免并行事项互相卡点；
- ⑥ 负责把外部世界的信息（市场、合规进度、供应链条款）带回团队，同时是资质代办机构的对接人；
- 公司注册、算法备案、ICP 许可证等专业事项，建议由 ① 拍板 + ⑥ 执行对接代办机构，不建议团队自行摸索走流程以免延误整体节奏；
- 6 人团队规模较小，不单独设置全职 QA/客服角色，建议②③④⑤在测试、内容审核等阶段性工作上互相支援。

## 七、预算与风险提示

**主要成本项（量级参考，需按实际情况询价）**：
- 公司注册代办 + 商标注册：数千至一万元级
- 生成式AI算法备案代办：数万元级，周期 1-3 个月
- 增值电信业务许可证（如需要）：对注册资本、场地、人员有硬性要求，代办费用数万元级
- 软件著作权登记：数千元/项
- 企业开发者账号年费：Apple 299 美元/年 + 各安卓商店认证费用
- 服务器/数据库/AI API 调用成本：随用户量线性增长，需提前做成本模型
- 供应链合作保证金/技术对接成本：视合作方而定

**主要风险**：
1. **6 个月压缩节奏对备案审批的容错空间很小** —— 生成式AI备案或小程序审核只要被打回一次，就可能导致整体延后至 7-8 个月；建议预留 Buffer，并准备低配备用方案（如先以"人工客服+规则引擎"的低配版本上线，AI 能力备案通过后再灰度开放）；
2. **生成式AI备案周期不确定** —— 必须在 W1 立即启动，作为整体上线节奏的关键路径；
3. **宠物食品类资质复杂度高** —— 早期用联营模式规避，避免拖慢上线；
4. **小程序类目审核不通过风险** —— 提前按官方最新类目要求准备资质，建议选择资质门槛最低的类目起步；
5. **AI 模型单点依赖风险** —— 尽早做多模型容灾；
6. **一人公司过渡期的股权/治理风险** —— 需在第 1-2 个月内完成增资扩股并签署合伙人协议，避免长期停留在一人公司架构；
7. **团队/资金规模风险** —— 6 人团队身兼多职，且节奏压缩后并行事项多，需优先外包法务/代办类工作，聚焦团队精力在产品与增长上。
